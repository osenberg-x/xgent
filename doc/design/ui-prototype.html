理清了。XGent 的 steering 机制分两层：
📖 steering race 机制rust
// 内层 stream_llm_response：流式期即时中断
loop {
    tokio::select! {
        chunk = stream.next() => { // 正常累加 TextDelta }
        steer = steering_rx.recv() => {
            // race 到 steering → 即时 cancel 当前流
            cancel.cancel();
            return StreamOutcome {
                pending_steering: Some(steer), ..
            };
        }
    }
}
// → run_agent_loop 收到 pending_steering，注入 req.messages 后重新流式
// → ECS 侧同步 conv.push_user，UI 展示 + 持久化
核心是两点：
📖 停止边界补偿rust
// 外层循环停止边界重新轮询
if let Ok(steer) = steering_rx.try_recv() {
    req.messages.push(steer.into_message());
    continue; // 不 break，重新进入内层
}
**机制要点**： 1. **流式期即时中断** — `stream_llm_response` 用 `tokio::select!` 同时监听 SSE chunk 与 steering channel，steering 到达立即 cancel 当前流、返回 `pending_steering`，由 `run_agent_loop` 注入 `req.messages` 后重新流式。 2. **停止边界补偿轮询** — 外层循环在每个 yield 点（内层返回时）用 `try_recv` 重新检查 steering，防止 steer 在 agent 准备停止时丢失。 3. **ECS 同步** — steering 期间 UI 侧 `conv.push_user` 把用户的补充输入展示出来并持久化，对齐 omp `streamAssistantResponse` 的 abort race 语义。 这设计对齐了 omp 的 `streamAssistantResponse`：流式不阻塞 steering，steering 即时打断流而非排队等待完成。
