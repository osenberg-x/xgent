# XGent 需求设计文档

> 本文档从零定义 XGent 的产品需求。不参考任何旧有设计文档，作为项目需求的权威来源。
>
> 状态：草案 v2 · 待评审

---

## 1. 项目定位与目标

### 1.1 定位

**XGent 是一个面向个人开发者日常编码的桌面端 AI Code Agent 工具，基于 Bevy 游戏引擎构建。** 它以一个完整可用的产品为目标，让开发者在本地项目中获得 AI 辅助编码、代码理解、自动化操作的能力，同时通过轻量创意的视觉与陪伴体验，区别于同类工具。

### 1.2 核心价值主张

- **实用优先**：覆盖一个 AI code agent 的基本完整功能，能真正用于日常编码，不是技术试验品。
- **数据驱动体验**：利用 Bevy ECS 与数据驱动 UI，提供比同类工具更顺滑、更可推理、更可测试的交互。
- **创意陪伴**：通过桌面虚拟宠物等情感化设计，让工具不只是冷冰冰的命令面板。
- **可扩展**：在 TUI、Web、3D 可视化、自定义工具等方向上预留架构空间，前期不实现但不阻塞。

### 1.3 产品形态

- **当前形态**：跨平台桌面 GUI 应用（Windows / macOS / Linux）。
- **预留形态**：
  - TUI 前端（未来作为同一套核心的另一套 UI 前端实现）。
  - Web 端（架构层面留好抽象接口，当前不实现）。
  - 3D 可视化（作为可选插件层，MVP 不实现、不阻塞主线）。

## 2. 目标用户与使用场景

### 2.1 目标用户

个人开发者，在本地编码项目中希望借助 AI 完成日常编码、代码理解、自动化操作的开发者。

### 2.2 使用场景

- 在本地代码项目中与 AI 多轮对话，请求修改、解释、生成代码。
- 让 AI 通过工具读写文件、搜索代码、运行命令，并经用户确认后执行高危操作。
- 同时并行处理多个项目（多开窗口）。
- 在编码陪伴中获得轻量的情感化反馈（桌面宠物）。

## 3. 设计原则

| 原则 | 含义 |
|:---|:---|
| **实用优先** | 任何创意/视觉服务于实用性，不为酷炫牺牲核心流程效率。 |
| **数据驱动** | UI 是 Agent 状态的实时投影；子系统经抽象事件/消息通信，可推理、可测试、可回放。 |
| **可扩展** | TUI / Web / 3D / 自定义工具在架构上预留，但不引入当前不用的复杂度。 |
| **轻量** | 多开、共享运行时资源、低占用；不因游戏引擎背景而忽视资源效率。 |
| **3D 留白** | 3D 能力作为可选插件层存在，MVP 不实现、不阻塞主线。 |
| **跨平台** | 桌面端 Windows/macOS/Linux 同步支持；为未来 Web 端留接口。 |

## 4. 功能需求

功能逐条编号，标注优先级。优先级含义：
- **MVP**：最小可用版本必须包含。
- **P1**：MVP 之后第一批迭代。
- **P2**：后续迭代，架构需考虑但不实现。

### 4.1 核心功能（MVP）

| 编号 | 功能 | 描述 | 优先级 |
|:---|:---|:---|:---|
| F-01 | 多轮对话 | 与 AI 进行多轮上下文对话，支持中断、重试。 | MVP |
| F-02 | 流式输出 | AI 回复以流式方式实时呈现。 | MVP |
| F-03 | 工具调用 | AI 可调用工具：读文件、写文件、搜索代码、运行命令。 | MVP |
| F-04 | 操作确认 | 写文件、运行命令等高危操作需用户确认后执行；只读类操作自动放行。 | MVP |
| F-05 | 项目上下文 | 感知当前项目结构、读取并理解项目代码。MVP 采用无索引·按需读取（A）；编辑器上线后按 OQ-08 演进升级。 | MVP |
| F-06 | 会话管理 | 新建、切换、查看历史会话。 | MVP |
| F-07 | Provider 切换 | 支持本地模型（Ollama 等）与云端（OpenAI / Anthropic 等）多 provider；支持第三方兼容接口（OpenAI compatible 等）与新的响应式接口（如 Response API）；支持自定义 API；运行时切换。 | MVP |
| F-08 | 命令面板 | 类似 VSCode 的快速命令/文件入口（Cmd+P / Cmd+Shift+P 风格）。 | MVP |
| F-09 | 快捷键体系 | 参考 VSCode 的快捷键体系作为基准，覆盖命令面板之外的常用操作。 | MVP |

### 4.3 虚拟宠物（F-15 详述）

**定位**：纯情感陪伴，不影响任何功能能力，不与 agent 能力挂钩。可整体开关。

**机制设计**：
- **单只宠物**：每个用户一只，长期养成。
- **等级系统**：随用户使用时长增长，宠物等级提升；等级纯粹解锁外观/形态变化，不影响 agent 功能。
- **Agent 状态映射**：宠物的表情、动作、台词反映 agent 当前状态（如：思考中 / 执行工具中 / 完成 / 出错），作为状态的可视化提示，但不改变 agent 行为。
- **外观来源**：提供一只默认宠物；支持用户上传自定义外观。
- **对话陪伴**：宠物作为与用户对话/需求的存在感载体（具体呈现形式待定，不取代 F-01 的主对话流）。
- **桌面渲染**：以置顶透明无边框窗口悬浮于桌面其他窗口之上（类似桌面宠物）。
- **开关**：宠物功能支持整体开关；关闭后完全不加载、不占用资源。

**优先级**：P1（MVP 之后第一批迭代）。MVP 不实现宠物，但架构需确保宠物可作为独立可选模块加载/卸载。

## 5. 非功能需求

| 编号 | 需求 | 说明 |
|:---|:---|:---|
| NF-01 | 跨平台 | Windows / macOS / Linux 桌面端同步支持。 |
| NF-02 | 轻量多开 | 支持同时开多个项目，每个项目一个窗口；多开时共享全局配置与 provider 进程/模型连接等运行时资源，尽量降低占用；项目配置隔离；同一项目开多个客户端时需共享同步文件状态。 |
| NF-03 | 性能 | UI 零延迟响应；核心编辑/对话流程不被渲染或动画阻塞。 |
| NF-04 | 可维护性 | 模块边界清晰，每个子系统可独立测试；支持无渲染的 headless 运行与消息流录制/回放。 |
| NF-05 | 国际化 | 从一开始就考虑国际化（i18n）架构，所有用户可见字符串走 i18n；前期以中文为主。 |
| NF-06 | 可扩展 | TUI / Web / 3D / 自定义工具 / MCP 的接入点在架构上预留，不因当前不实现而阻塞未来扩展。 |

## 6. 关键约束

1. **宠物纯情感向**：虚拟宠物（F-15）不影响任何功能能力，等级仅解锁外观，不改变 agent 行为。宠物功能支持开关，关闭后不占用资源。
2. **共享运行时降占用**：GUI 多开时，provider 进程/模型连接等运行时资源需共享，避免每窗口独立占用。
3. **Web 接口预留**：provider / tools / 文件访问等需走抽象层，使 Web 端（受 WASM 限制无法直接本地文件/子进程）将来可注入不同实现。当前不实现 Web，但架构不得与 Web 形态冲突。
4. **3D 作为可选插件层**：3D 可视化（F-16）作为可选插件层存在，MVP 不实现、不阻塞主线。
5. **TUI 架构留口**：前期仅 GUI，TUI 不实现；核心逻辑与 UI 前端解耦，使未来 TUI 作为同一套核心的另一套前端实现，且不影响当前性能。
6. **沿用项目名**：继续使用 XGent，待有更合适名称再更名。
7. **参考成熟项目实现**：对于项目上下文检索（F-05）等需深入设计的功能点，需参考成熟项目实现后再定策略，不闭门造车。

## 7. 分期路线

### 7.1 MVP（最小可用版本）

目标：一个真正能日常使用的 AI code agent GUI 桌面应用。

范围：
- F-01 多轮对话
- F-02 流式输出
- F-03 工具调用（读/写/搜/运行）
- F-04 操作确认
- F-05 项目上下文
- F-06 会话管理
- F-07 Provider 切换（本地 + 云端 + 自定义 API + 兼容/响应式接口）
- F-08 命令面板
- F-09 快捷键体系（参考 VSCode）
- NF-01 跨平台
- NF-02 轻量多开
- NF-03 性能基线
- NF-04 可维护性基线

**不含**：宠物、3D、TUI、Web、Git、内置编辑器、成本统计、MCP、自定义工具。

### 7.2 第一批迭代（P1）

- F-10 Git 集成（优先级可放低）
- F-11 内置编辑器（上线后触发 OQ-08 检索升级：C → D → E）
- F-12 成本统计
- F-13 MCP 支持
- F-15 虚拟宠物

### 7.3 后续迭代（P2）

- F-14 自定义工具
- F-16 3D 可视化
- F-17 TUI 前端
- F-18 Web 端

## 8. 术语表

| 术语 | 含义 |
|:---|:---|
| Agent | 承载对话循环与工具调用的核心引擎。 |
| Provider | LLM 提供方，如 OpenAI / Anthropic / Ollama。 |
| 工具 | Agent 可调用的能力，如读文件、写文件、搜索、运行命令、MCP 工具。 |
| 命令面板 | 全局快速命令/文件入口，类似 VSCode 的 Cmd+P / Cmd+Shift+P。 |
| 兼容接口 | OpenAI compatible 等第三方接口，复用 OpenAI 协议格式。 |
| 响应式接口 | 如 Response API 这类新型响应接口，区别于传统 chat completions。 |
| i18n | internationalization，国际化；从架构上让所有用户可见字符串可多语言化。 |
| 会话 | 一次完整的多轮对话上下文。 |
| Provider 进程共享 | 多开窗口共用同一 provider 运行时与模型连接，避免重复占用。 |
| 桌面宠物 | 以置顶透明窗口渲染于桌面的虚拟宠物，纯情感陪伴。 |
| MVP | 最小可用版本，本项目的第一个交付里程碑。 |

## 9. 开放问题

以下点已初步决策但实现细节待后续评估，不阻塞本需求文档定稿：

- **OQ-05**：宠物“对话陪伴”（F-15）的具体呈现形式——宠物气泡？独立面板？不取代主对话流的边界在哪。当前仍不确定，留待 P1 阶段设计时再定。
- **OQ-07**：会话持久化形态——本地文件、格式、是否跨项目。留待实现时考虑评估。
- **OQ-10**：成本统计（F-12）的数据来源与展示形态。留待实现时考虑。

以下点已决策并落入正文，仅作记录：

- **OQ-01（已决策）**：支持 OpenAI / Anthropic / Ollama 等，及第三方兼容接口（OpenAI compatible）、新的响应式接口（如 Response API）、自定义 API。见 F-07。
- **OQ-02（已决策）**：内置编辑器为中等能力边界（多行+行号+undo/redo+查找+tree-sitter 高亮，MVP 仅 Rust）；基于 bevy_ui 自造（`xui::CodeEditor`，可独立发布），egui 混合方案作 P1+ 备选。完整能力边界（+LSP+split view）留后续。见 F-11 与 `doc/design/editor-design.md`。
- **OQ-03（已决策）**：快捷键参考 VSCode 体系。见 F-09。
- **OQ-04（已决策）**：默认一只宠物 + 用户上传自定义外观。见 4.3。
- **OQ-06（已决策）**：共享全局配置，项目配置隔离；同一项目多客户端共享同步文件状态。见 NF-02。
- **OQ-08（已决策）**：采用演进式路径——MVP 与编辑器上线前用选项 1（A 无索引·按需读取 → B tree-sitter repo map）；**内置编辑器（F-11）上线触发到 C（向量 RAG）**，D（LSP/AST）**延后**到 LSP 真正接入时（即编辑器升完整边界），E（混合检索）跟随 D。分段理由：中等边界不含 LSP，D 依赖 LSP，解耦避免编辑器上线就硬依赖 LSP。索引层与 agent 之间走抽象接口，未来 Web 端可注入不同实现。详见 `doc/notes/context-retrieval-research.md` 与 `doc/decisions/0010-oq08-检索升级路径分段-编辑器到c-d延后.md`。
- **OQ-09（已决策）**：国际化从一开始就考虑。见 NF-05。
