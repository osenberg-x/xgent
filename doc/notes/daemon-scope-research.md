# 守护进程胖瘦方案调研

> 本文档对比 XGent 多进程架构下"后台守护进程"的职责范围方案，供决策参考。
> 前提：已确定多进程方案 B（每个项目/窗口一个独立 UI 进程，共享一个后台守护进程）。
> 状态：调研报告 · 待决策

---

## 0. 背景

需求 NF-02 要求：多开共享 provider 进程、共享全局配置、项目配置隔离、同项目多客户端共享同步文件状态。

已选定**多进程方案 B**：每个项目/窗口一个独立 UI 进程（xgent-ui），共享一个轻量后台守护进程（xgent-daemon）。本调研对比守护进程的职责范围（胖瘦）。

---

## 1. 守护进程候选职责

从需求反推出后台可能承担的职责：

| 编号 | 职责 | 来源需求 | 是否必须由后台承担 |
|:---|:---|:---|:---|
| a | Provider 连接池/模型连接共享 | NF-02 多开共享 provider 进程 | ✅ 是（多进程下必然） |
| b | 全局配置读写协调 | NF-02 全局配置共享 | ✅ 是（多进程需单一写入点或文件锁） |
| c | 文件系统监听（项目内文件变更） | F-05 实时性、NF-02 同项目多客户端共享文件状态 | ✅ 强烈建议（多客户端需统一事件源，避免各自监听重复且状态不一致） |
| d | 同项目多客户端文件状态同步 | NF-02 | ✅ 是（必须有一个协调点） |
| e | 检索/索引维护（A→B→C 阶段） | OQ-08、F-05 | ⚠️ 候选（B/C 阶段需持续索引，放后台更合理） |
| f | 会话历史持久化 | F-06 | ⚠️ 候选（可放后台统一，也可各客户端本地） |
| g | 工具执行（读写文件/运行命令） | F-03、F-04 | ⚠️ 候选（放后台可统一授权与审计） |
| h | 成本统计聚合 | F-12 | ⚠️ 候选（多客户端聚合需统一收口） |

---

## 2. 三种胖瘦方案

### 方案 1：瘦后台

**职责**：a + b + c + d（provider 连接、全局配置、文件监听、多客户端文件状态同步）

**UI 进程承担**：e/f/g/h 全部——每个 UI 进程各自维护索引、各自持久化会话、各自执行工具、各自统计成本。

**优点**：
- 后台极简，职责清晰，IPC 消息量小（只有 provider 请求/响应、配置读写、文件变更通知）。
- MVP 实现成本最低。
- 工具执行在 UI 进程，用户授权确认就近，响应快。
- 单点故障影响面小：后台崩溃只丢 provider 连接与文件同步，会话/工具仍可本地工作。

**缺点**：
- 同项目多客户端的索引/会话/成本各自维护，**易不一致**（A 方案无索引时影响小；升到 B/C 阶段，多客户端各建索引是浪费且可能不一致）。
- 工具执行分散，难统一审计/限流。
- 多开时每个 UI 进程都要做索引，资源占用随开窗数增长。

**适合**：MVP（A 无索引阶段）；多开客户端数少；对一致性要求不高。

---

### 方案 2：中后台

**职责**：瘦后台 + e + f（加索引维护、会话历史持久化）

**UI 进程承担**：g（工具执行）+ h（成本统计）仍在 UI 侧

**优点**：
- 索引与文件监听合一：后台统一监听文件变更 → 增量更新索引 → 推送给订阅该项目的 UI 进程。**多客户端共享同一份索引**，无重复构建，状态一致。
- 会话历史统一持久化，多客户端可看到同项目的所有会话（符合"共享同步文件状态"精神延伸到会话）。
- 多开资源占用更优：索引只建一次。
- 为 C/D/E 检索升级铺路：向量库/LSP 索引天然适合放后台长驻。

**缺点**：
- 后台复杂度上升（需内建或嵌入索引引擎、向量库、tree-sitter）。
- 索引冷启动：首次开项目后台需建索引，UI 进程需等待/进度反馈。
- 工具执行仍在 UI 侧，多客户端工具调用仍各自授权、无统一审计。
- MVP（A 阶段无索引）时，中后台相对瘦后台无明显收益，但复杂度已上去了。

**适合**：进入 B 阶段（repo map）之后；多开同项目场景多；追求一致性。

---

### 方案 3：胖后台

**职责**：中后台 + g + h（加工具执行、成本统计聚合）

**UI 进程承担**：只做 UI 渲染与交互、对话编排（编排仍可能放 UI 侧或后台，见下"补充问题"）

**优点**：
- 工具执行统一在后台，**多客户端工具调用统一授权/审计/限流**，安全性最高。
- 文件读写经后台统一收口，同项目多客户端文件状态强一致（避免两个客户端同时改同一文件冲突）。
- 成本统计天然聚合（多客户端/多 provider 调用统一记账）。
- UI 进程极轻，崩溃/重启不丢会话与索引。
- 最接近"服务端化"形态，未来 Web 端（F-18）几乎可直接复用后台。

**缺点**：
- 后台最复杂，IPC 消息量最大（每次工具调用、确认、结果都跨进程）。
- 用户授权确认跨进程：工具需确认时，后台→UI 弹窗→用户决策→回传后台执行，链路长、延迟略增、复杂度高（需异步确认协议）。
- 单点故障影响面最大：后台崩溃则所有 UI 窗口失去工具能力。
- MVP 实现成本最高。
- 隐私/权限边界：后台执行子进程（RunCommand）意味着后台进程权限可能高于 UI，需谨慎设计权限模型。

**适合**：产品成熟期；多客户端协作高频；强安全审计需求；为 Web 端铺路。

## 3. 对比矩阵

| 维度 | 方案 1 瘦后台 | 方案 2 中后台 | 方案 3 胖后台 |
|:---|:---|:---|:---|
| 后台职责范围 | a,b,c,d | +e,f | +g,h |
| 后台复杂度 | 低 | 中 | 高 |
| IPC 消息量 | 小 | 中 | 大 |
| 多开资源占用 | 随开窗增长（索引/会话各自） | 优（索引/会话共享） | 最优（工具也共享） |
| 单点故障影响面 | 小 | 中 | 大 |
| MVP（A 阶段）实现成本 | 最低 | 中（为未到的索引付代价） | 高 |
| B/C 阶段一致性 | 差（多客户端各自索引） | 优 | 优 |
| 工具统一授权/审计 | 无 | 无 | 有 |
| Web 端复用度 | 低 | 中 | 高 |
| 演进友好度 | 需重构才能升 | 自然演进到胖 | 起点 |

## 4. 演进视角

三种方案并非互斥，可随阶段演进：

```
MVP（A 检索）：方案 1 瘦后台
  ↓ B 阶段（repo map）：索引职责上移 → 方案 2 中后台
  ↓ C/D 阶段 + 编辑器上线：工具执行是否上移视安全需求 → 视情况升方案 3
  ↓ Web 端：若要做，后台需承担工具执行 → 方案 3
```

关键判断点：
- **B 阶段（repo map）是切换到中后台的自然时机**——因为索引首次出现，多客户端共享索引收益明显。
- **Web 端是切换到胖后台的强制时机**——WASM 无法本地文件/子进程，工具执行必须经后台/服务端。

## 5. 候选决策选项

| 选项 | MVP | B 阶段 | C/D 阶段 | 说明 |
|:---|:---|:---|:---|:---|
| **选项 1（推荐）** | 方案 1 瘦后台 | 升 方案 2 中后台 | 视安全/Web 需求升 方案 3 | 演进式，MVP 最轻，自然升级路径 |
| **选项 2** | 方案 2 中后台 | 同 | 升 方案 3 | MVP 即上中后台，为索引付提前代价，但 B 阶段无需重构 |
| **选项 3** | 方案 3 胖后台 | 同 | 同 | MVP 即全功能后台，最重但一步到位、Web 友好 |
| **选项 4** | 方案 1 瘦后台 | 升 方案 2 | 不升胖（工具永留 UI 侧） | 轻量到底，放弃统一审计与 Web 端工具复用 |

---

## 6. 我的倾向（仅供参考，非决策）

倾向**选项 1**：

- MVP 用瘦后台，把精力集中在 provider 共享、配置、文件同步这些真正决定"多开能用"的部分。
- B 阶段（repo map）是中后台的自然切换点，索引上移有真实收益且不返工。
- 胖后台推迟到"Web 端要做"或"统一审计成刚需"时再升，避免 MVP 背上跨进程授权协议的复杂度。
- 关键：后台职责用 trait 抽象，e/f/g/h 都设计成"可上移可下放"的接口，使演进不推翻架构。

但**最终决策由你定**。

## 7. 补充问题（无论选哪个都需明确）

1. **对话编排（agent loop）放哪侧**：UI 进程还是后台？
   - 放 UI 侧：每个客户端独立对话循环，简单、隔离好；但多客户端同项目对话不共享。
   - 放后台侧：对话循环集中，可多客户端共享/协调；但后台更胖、更复杂。
   - 我倾向 UI 侧（MVP 每客户端独立 agent loop），后台只做"资源池"。

2. **IPC 协议选型**：本地 Unix socket / named pipe / gRPC / 自定义 JSON-RPC？
   - 影响：跨平台、性能、可调试性、未来 Web 端复用度（gRPC/JSON-RPC 更易复用）。

3. **后台生命周期**：随第一个 UI 进程启动、随最后一个退出？还是常驻系统服务（开机自启）？

这三个我会在你选定胖瘦方案后一并确认。
