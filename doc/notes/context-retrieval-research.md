# 项目上下文检索方案调研

> 本文档调研当前主流 AI code agent 的项目上下文检索方案，给出分类、代表项目、优缺点对比，供 XGent 决策（OQ-08）参考。
>
> 状态：调研报告 · 待决策

---

## 0. 背景

XGent 需要在 MVP 中实现 F-05「项目上下文」：让 agent 感知当前项目结构、读取并理解项目代码。核心问题是：**当用户提问时，如何决定把哪些代码/上下文喂给 LLM？**

检索方案直接决定 agent 的能力上限、响应速度、本地资源占用、隐私边界、实现复杂度。本调研梳理主流方案供决策。

---

## 1. 方案分类总览

| 方案 | 代表项目 | 是否需索引 | 索引粒度 | 实时性 | 本地化 | 大库支持 | 实现复杂度 |
|:---|:---|:---|:---|:---|:---|:---|:---|
| A. 无索引·按需读取 | Aider、Claude Code、Codex CLI | 否 | 无 | 实时 | ✅ | 差 | 极低 |
| B. 文件树 + 语义检索树 | Aider（repo map） | 是（轻量） | 符号/token | 准实时 | ✅ | 中 | 低 |
| C. 向量嵌入检索（RAG） | Cody、Continue、Cursor（部分）、Cursor 早期 | 是 | chunk 级 | 需重建 | 视部署 | 优 | 中 |
| D. AST/符号级检索 | Sourcegraph、LSPI多 | 是 | 符号级 | 实时（LSP） | ✅ | 优 | 中高 |
| E. 混合（向量 + 符号 + 关键词） | Cursor（当前）、Continue（混合）、Cody（混合） | 是 | 多粒度 | 增量 | 视部署 | 优 | 高 |
| F. 云端托管索引 | Cursor、GitHub Copilot | 是 | chunk 级 | 后台 | ❌ | 优 | 客户端低 |

---

## 2. 各方案详述

### 方案 A：无索引·按需读取

**做法**：不预建任何索引。用户提问后，agent 通过树状目录结构 + grep/ripgrep + 按需读取文件，像人一样"翻代码"。典型流程：列目录 → 读关键文件 → grep 关键词 → 读相关文件 → 组织上下文送 LLM。

**代表**：Aider（早期无 map 时）、Claude Code、Codex CLI、大部分"终端 agent"。

**优点**：
- 实现极简，无需向量数据库、嵌入模型。
- 永远实时，文件改动立即可见，无索引失效问题。
- 隐私好，全本地，零数据外泄。
- 无冷启动，打开即用。

**缺点**：
- 大库差：每次都要现搜，token 浪费在"找文件"上，慢且易漏。
- 无"语义"匹配，靠 grep 关键词，问"处理用户登录的逻辑在哪"这类语义查询无能为力。
- 上下文质量高度依赖 agent 的搜索策略 prompt，不稳定。

**适合**：中小项目（< ~50 文件密集改动场景）；隐私敏感场景；MVP 起步。

---

### 方案 B：文件树 + 语义检索树（repo map）

**做法**：预构建一个"仓库地图"：扫描所有文件，用 tree-sitter 解析出符号（函数/类/方法名）及其引用关系，按 token 预算压缩成一个 ranked map 喂给 LLM。LLM 据此知道"项目有哪些符号、谁引用谁"，再按需读文件。Aider 的 repo map 是典型。

**代表**：Aider（repo map，基于 tree-sitter）。

**优点**：
- 轻量：索引就是符号名 + 引用图，体积小（几十 KB ~ 几 MB）。
- 语义结构强：LLM 能看到项目骨架与依赖关系，做"架构级"理解。
- 本地化好，纯 CPU tree-sitter，无嵌入模型。
- 实时性好：文件改动后增量重解析单个文件即可。

**缺点**：
- 仍非"语义检索"：问"和支付相关的代码"时，靠符号名匹配，若命名不直白则失效。
- tree-sitter 需为每种语言加载 grammar，语言覆盖受限于可用 grammar。
- 大库的 map 仍可能超 token 预算，需截断。
- 不提供 chunk 级"相似度"匹配。

**适合**：代码为主、注重结构理解的项目；本地化优先；中等规模库。

---

### 方案 C：向量嵌入检索（RAG）

**做法**：把代码切成 chunk，每个 chunk 用嵌入模型生成向量，存入向量库。用户提问时把问题也嵌入，做相似度检索取 top-k chunk 喂给 LLM。

**代表**：Cody（早期）、Continue（默认）、Cursor（早期）、大量自建 RAG 工具。

**优点**：
- 真正语义匹配：能找到"含义相近"的代码，不靠命名。
- 大库友好：向量检索 O(log n)。
- chunk 级粒度，token 利用率高。

**缺点**：
- 嵌入模型成本：本地需跑嵌入模型（如 all-MiniLM、bge）；云端则隐私差。
- 索引冷启动：首次索引大库耗时（分钟级）。
- 实时性差：文件改动需重新嵌入该 chunk，否则检索到旧内容；全量重建慢。
- 嵌入模型对"代码语义"理解有限，常不如纯关键词检索（BM25）精准。
- 实现复杂：向量库选型（本地如 lancedb/qdrant；嵌入服务选型）。

**适合**：大库；语义查询多；可接受冷启动与增量维护成本。

---

### 方案 D：AST/符号级检索（LSP 驱动）

**做法**：用 Language Server Protocol 或 tree-sitter 构建符号索引（定义、引用、类型），检索时按符号/引用关系找代码，类似 IDE 的"Find References""Go to Definition"。Sourcegraph 的代码搜索是代表。

**代表**：Sourcegraph、LSPI（IDE 内）、部分企业内部工具。

**优点**：
- 精准：符号引用关系是最可靠的"代码关联"，无歧义。
- 实时：LSP 本就跟踪文件变更，索引随改随更新。
- 本地化好（本地 LSP）。
- 强结构理解：rename、调用链、类型层级都能做。

**缺点**：
- LSP 集成复杂：每种语言一个 LSP server，启动/通信/能力差异大。
- LSP 主要为编辑器设计，做"全库语义检索"需扩展。
- 非"语义模糊查询"强项（问"处理支付的代码"不如向量）。
- 实现成本高。

**适合**：重 IDE 体验、强结构操作的项目；有 LSP 运维能力的团队。

---

### 方案 E：混合检索（向量 + 符号 + 关键词）

**做法**：多路召回——向量（语义）+ BM25/ripgrep（关键词）+ 符号/AST（结构），各取 top-k 后融合排序（reciprocal rank fusion 等）送 LLM。当代生产级 agent 多走此路。

**代表**：Cursor（当前）、Continue（混合模式）、Cody（混合）。

**优点**：
- 召回最全：语义、词面、结构三路互补，覆盖各类查询。
- 生产级效果最好。
- 鲁棒：单路失效仍有其他路兜底。

**缺点**：
- 实现最复杂：三套索引 + 融合排序 + 增量同步。
- 资源占用最高：向量库 + 关键词索引 + LSP/AST。
- 维护成本高。

**适合**：成熟产品追求最佳效果；有工程投入；不在 MVP。

---

### 方案 F：云端托管索引

**做法**：客户端不建索引，代码上传云端建索引与检索，客户端只发查询。Cursor 与 GitHub Copilot 走此路。

**代表**：Cursor、GitHub Copilot。

**优点**：
- 客户端极轻，零本地索引成本。
- 效果可做到最好（云端有强模型 + 专有索引算法）。
- 大库无压力。

**缺点**：
- 隐私差：代码需上云，与 XGent 的"本地优先、支持本地模型"定位冲突。
- 强依赖网络与服务稳定性。
- 与"支持本地模型、自定义 API"理念相悖。

**结论**：**与 XGent 定位冲突，不采用。** 列入仅供对比。

## 3. 关键决策维度对比

### 3.1 本地化与隐私

| 方案 | 本地化 | 嵌入模型需求 | 数据外泄 |
|:---|:---|:---|:---|
| A 无索引 | ✅ 全本地 | 无 | 无 |
| B repo map | ✅ 全本地 | 无（tree-sitter） | 无 |

XGent 强调本地优先、支持本地模型，**F 排除**，C/E 需配套本地嵌入方案。

## 4. 演进路径建议

调研发现主流项目的共同演进规律：**几乎都从 A/B 起步，验证可用后逐步加 C/D，最终走向 E**。

- Aider：A → B（repo map），保持本地轻量。
- Cody/Continue：起步即 C，后加混合。
- Cursor：C → E，加云端专有索引。

对 XGent 而言，关键约束：本地优先、跨平台、轻量多开、MVP 快速可用、未来 Web/TUI 扩展、参考成熟项目。**演进式路径**最契合：

```
MVP：A（无索引·按需读取 + ripgrep + 目录树）
  ↓ 验证可用后
P1：B（tree-sitter repo map，加结构理解）
  ↓ 按需
P2：C/D 之一或 E（向量或 LSP，按实际痛点与资源决定）
```

理由：
- MVP 用 A，零冷启动、零依赖、隐私无虞、实现极快，先把对话与工具链跑通。
- B 是 A 的自然增强，仍全本地、轻量，token 利用率显著提升。
- C/D/E 留到有了真实使用数据再定，避免过度工程。

## 5. 候选决策选项（供你选定 OQ-08）

| 选项 | MVP 用 | P1 升级 | P2 升级 | 说明 |
|:---|:---|:---|:---|:---|
| **选项 1（推荐）** | A 无索引·按需读取 | B tree-sitter repo map | C 向量 RAG 或 D LSP | 演进式，本地优先，MVP 最快可用 |
| **选项 2** | A 无索引·按需读取 | B repo map | E 混合 | 同上，P2 直接奔生产级混合 |
| **选项 3** | B tree-sitter repo map | C 向量 RAG | E 混合 | MVP 即上结构索引，起点高、复杂度略增 |
| **选项 4** | C 向量 RAG（本地嵌入） | E 混合 | — | MVP 即语义检索，最强但最重、有冷启动 |
| **选项 5** | A 无索引·按需读取 | D LSP/AST | E 混合 | 走 IDE 风格，重结构操作 |

各选项的共同前提：
- 均排除云端托管（F），与本地优先定位一致。
- 嵌入模型若引入，优先本地（如 Ollama 可跑嵌入，或本地 bge/MiniLM），避免数据外泄。
- 索引层与 agent 之间走抽象接口，使未来 Web 端可注入不同实现。

## 6. 我的倾向（仅供参考，非决策）

倾向**选项 1**：

- MVP 阶段把精力集中在对话循环、工具调用、provider、UI 这些真正决定"能不能用"的部分，检索用 A 足够跑通。
- Aider 的实践证明，仅靠 A/B 在中小项目上效果已经相当可用，不必 MVP 就背上向量库。
- 留好抽象接口，未来加 B/C/E 都不会推翻架构。

但**最终决策由你定**。如果你的使用场景以超大库为主，或语义检索是刚需，可考虑选项 3/4。
