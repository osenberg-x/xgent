# i18n 方案调研

> 本文档对比 Rust 生态的国际化（i18n）方案，供 XGent 决策 D-01 参考。
> 状态：调研报告 · 待决策

---

## 0. 背景与约束

XGent 要求（NF-05）：**从一开始就考虑 i18n**，所有用户可见字符串走 i18n 层；前期以中文为主，但架构保证可翻译；未来支持多语言切换。

关键约束：
- 从一开始内置（不是后期补丁）。
- Bevy 全栈 UI，字符串来自 bevy_ui 渲染。
- 轻量（多开低占用，i18n 不能成为负担）。
- 运行时切换语言（用户可切，UI 实时刷新）。
- 主要服务"应用 UI 字符串"，不是大规模文档翻译。

---

## 1. 候选方案

### 方案 A：fluent（Mozilla 官方）

**简介**：Mozilla 的本地化系统，ICU MessageFormat 规范，Firefox、Rust 编译器自身错误信息都用。Rust 绑定 `fluent-rs`，运行时库 `fluent-bundle`，模板宏 `fluent-template`。

**资源格式**：`.ftl` 文件，每语言一个目录。

**示例**：
```ftl
# locales/zh-CN/main.ftl
welcome = 欢迎，{ $name }！
files-count = 文件数：{ $count }
items = { $count ->
    [one] 1 项
   *[other] { $count } 项
}
```

**优点**：
- 设计完备：复数、性别、选择、格式化、术语引用全支持。
- Mozilla 维护，规范成熟，工具链完善（pontoon 等）。
- 运行时切换语言天然支持（重新加载 bundle）。
- 翻译资源是纯文本，对翻译协作者友好。

**缺点**：
- 学习曲线：`.ftl` 语法、MessageFormat 表达式需学。
- 依赖稍多（fluent-rs + unic-langid + intl-memoizer 等）。
- 二进制体积略增（ICU 数据）。
- 与 Bevy 集成需自建一层 Resource 包装。

**适合**：长期多语言、专业本地化、需要复数/性别等高级特性。

---

### 方案 B：rust-i18n（编译期宏）

**简介**：`rust-i18n` crate，编译期宏 `t!`， YAML/TOML/JSON 翻译文件，运行时轻量查表。

**资源格式**：`locales/zh-CN.yml`、`locales/en-US.yml`。

**示例**：
```yaml
# locales/zh-CN.yml
welcome: 欢迎，{{name}}！
files_count: 文件数：{{count}}
```
```rust
t!("welcome", name = "xdo");
```

**优点**：
- 极简单：`t!` 宏 + YAML，零学习成本。
- 编译期加载，运行时是纯查表，零开销。
- 运行时切换语言支持（`rust_i18n::set_locale`）。
- 依赖轻。
- 对应用 UI 字符串足够。

**缺点**：
- 复数/性别支持有限（rust-i18n 有 `{{count}}` 简单复数但不如 fluent 强）。
- 翻译文件格式不如 fluent 规范化，大规模协作略弱。
- 宏在编译期加载文件，热加载需额外处理。

**适合**：中小型应用、UI 字符串为主、追求简单轻量。

---

### 方案 C：gettext 风格（pure_gettext / gettext-rs）

**简介**：经典 GNU gettext 工作流，`.po`/`.mo` 文件，`gettext("...")` 查表。

**优点**：
- 成熟规范，翻译行业熟悉。
- 工具链完备（poedit 等）。

**缺点**：
- 源码里写英文 msgid，中文是"翻译"，对"前期以中文为主"的开发流程别扭（要为每个字符串想英文 ID）。
- `.po` 格式繁琐。
- 运行时切换需 reload。
- 复数支持有但语法老式。
- Rust 绑定（gettext-rs）依赖 C 库，跨平台编译麻烦。

**适合**：已有 gettext 工作流的团队。**不适合 XGent**（中文为主 + 跨平台 + 轻）。

---

### 方案 D：自建轻量方案

**简介**：自建一个最简方案——key-value 资源（TOML/JSON），运行时查表，`tr("key", vars)` 函数。

**优点**：
- 零依赖、完全可控、最轻。
- 完全贴合 XGent 需求。
- 与 Bevy 集成自定义 Resource。

**缺点**：
- 复数/性别/格式化要自己实现，遇复杂本地化会欠账。
- 没有现成工具链与翻译协作。
- 重复造轮子，未来若需求增长需迁移。

**适合**：确信只需 UI 字符串查表、永不需复杂本地化的场景。

## 2. 对比矩阵

| 维度 | A fluent | B rust-i18n | C gettext | D 自建 |
|:---|:---|:---|:---|:---|
| 复数/性别/选择 | ✅ 完备 | ⚠️ 有限 | ✅ 老式支持 | ❌ 自实现 |
| 运行时切换 | ✅ | ✅ | ⚠️ reload | ✅ 自控 |
| 学习成本 | 中 | 低 | 中 | 低 |
| 依赖/体积 | 中 | 轻 | 重(C库) | 最轻 |
| 工具链/协作 | ✅ Mozilla | 一般 | ✅ poedit | 无 |
| Bevy 集成 | 需自建 Resource | 需自建 Resource | 别扭 | 自建 Resource |
| 跨平台编译 | ✅ | ✅ | ⚠️ C 库麻烦 | ✅ |
| 中文为主开发流 | ✅(key 是语义ID) | ✅(key 是语义ID) | ❌(msgid 是英文) | ✅ |
| 未来扩展性 | ✅ 强 | 中 | 中 | 弱 |
| 活跃度 | 高(Mozilla) | 中 | 中 | — |

## 3. 与 XGent 需求匹配分析

XGent 的 i18n 需求画像：
- 从一开始内置 → 任何方案都行。
- 前期中文为主 → A/B/D 都友好，C 不友好。
- UI 字符串为主 → B/D 足够，A 也能但偏重。
- 轻量多开低占用 → B/D 最优，A 中，C 差。
- 运行时切换 → A/B/D 都行。
- 未来可能多语言扩展 → A 最强，B 中，D 弱。
- 复数/格式化需求 → 应用 UI 字符串里复数会用到（如"N 个文件"），A 最强，B 够用，D 要自实现。

**关键权衡**：是否预期未来有多语言、专业本地化、复数/性别需求？
- 是 → fluent（A），一步到位、规范、可协作。
- 否，只 UI 字符串 → rust-i18n（B），轻、简、够用。
- 极致轻量、完全可控 → 自建（D），但欠扩展性。

## 4. 候选决策选项

| 选项 | 方案 | 说明 |
|:---|:---|:---|
| **选项 1** | A fluent | 规范、完备、可协作、未来扩展强；偏重但一步到位 |
| **选项 2** | B rust-i18n | 轻、简、够用；中等扩展性；UI 字符串为主最合适 |
| **选项 3** | D 自建轻量 | 最轻最控；扩展性弱；要自己兜复杂本地化 |
| **选项 4** | B 起步，未来视需求迁 fluent | MVP 用 rust-i18n 快速起步，架构抽象一层 `Localizer` trait，未来换 fluent 不破坏调用方 |

---

## 5. 我的倾向（仅供参考，非决策）

倾向**选项 4**：

- MVP 用 rust-i18n：轻、简、中文为主够用、零学习成本、不拖慢 MVP。
- 架构上抽象一层 `Localizer` trait（UI 侧只依赖 trait，不直接依赖 rust-i18n），使未来若需 fluent 的复数/协作能力可无缝替换。
- 既满足"从一开始内置 i18n"，又不为未到的复杂本地化付提前代价。

但若你预期 XGent 未来会有多语言社区协作（多语言翻译者参与），建议直接选 **选项 1 fluent**，避免迁移成本。
